# v5 연습모드/매칭 플로우 기획 검토 기록

## 배경

현재 UI는 왼쪽 탭과 화면 내부 버튼이 같은 목적지로 이동하는 구조다. 특히 연습모드와 매칭은 기능이 존재하지만, 사용자가 어떤 순서로 이동해야 하는지 이해하기 어렵다.

이 검토는 구현 전에 제품 플로우를 먼저 고정하기 위한 기록이다.

## 검토 역할

| 역할 | 검토 관점 | 결론 |
| --- | --- | --- |
| 서비스 기획 | 목적지와 진행 상태 구분 | `홈`, `로드아웃`, `연습`, `전적`만 직접 목적지로 두고 `매칭 중`, `전투 중`, `결과`는 진행 상태로 분리 |
| UX 기획 | CTA, 잠금 상태, 이동 흐름 | 홈 primary CTA를 하나로 줄이고, 매칭은 CTA 이후 자동 진행 상태로 처리 |
| 아트/UI 기획 | 게임 클라이언트형 시각 구조 | 연습은 카메라 중심, 전투는 전투 보드 중심, 매칭은 단계형 상태 표시 중심 |
| 시스템 기획 | 실제 구현 경계 | 연습은 local progress only, 매칭은 HTTP API와 WebSocket 기반 실제 플로우 |

## 확인한 현재 구현

- 연습모드는 현재 왼쪽 탭의 `연습모드`와 홈 내부 `연습모드 시작` 버튼으로 진입할 수 있다.
- 연습모드는 프론트 로컬 카메라 recognizer와 progress만 사용한다.
- 연습모드는 서버 전투 판정, 레이팅, 전적에 영향을 주지 않는다.
- 매칭은 `/api/v1/matchmaking/queue`와 WebSocket 이벤트를 사용하는 실제 플로우다.
- API 매칭 경로는 혼자 들어온 사용자를 연습 상대와 자동 매칭하지 않는다.

## 결정 사항

1. 왼쪽 탭은 직접 방문 가능한 목적지로만 제한한다.
2. 매칭, 전투, 결과는 탭에서 제거하고 상태 화면으로 전환한다.
3. 홈은 사용자 상태별 primary CTA를 하나만 제공한다.
4. 연습은 매칭 전 선택적 준비 모드로 정의한다.
5. 연습 화면은 카메라와 목표 sequence 진행률을 중심으로 재구성한다.
6. 매칭 화면은 큐 대기와 소켓 상태를 단계형으로 보여준다.
7. 전투와 결과는 서버 이벤트가 있을 때만 진입한다.
